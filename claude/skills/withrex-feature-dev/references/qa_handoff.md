# QA 핸드오프 가이드

## 완료 보고서 양식

```markdown
## 기능 개발 완료 보고

### 구현 요약
- 기능: [기능명]
- 변경된 파일: N개
- 새로 추가된 파일: N개

### 변경 내역
| 파일 | 변경 유형 | 설명 |
|------|----------|------|
| ... | 수정/신규/삭제 | ... |

### 생성된 테스트 시나리오
- 총 N개 시나리오 추가 (tests/withrex_테스트_스크립트.md)
- Backend: N개
- Frontend: N개
- Integration: N개

### 자체 검증 결과
- 통과: N개
- 수정됨: N개 (상세: .claude/bug/bug_update.md 참조)

### QA 검증 권장 범위
- 카테고리: [이 기능과 관련된 카테고리 코드]
- 모드: 스캔모드 / 취약점모드 / 둘 다
- 특이사항: [간접 영향 파일 등 주의할 부분]
```

---

## QA 파이프라인 호출 방법

완료 보고서를 전달한 후, 사용자에게 아래 지시문을 제안한다:

```
.claude/skills/withrex-qa-pipeline/SKILL.md 스킬을 읽고,
[기능명] 관련 테스트 시나리오를 검증해줘.

테스트 범위:
- 카테고리: [관련 카테고리 코드]
- 모드: [스캔모드/취약점모드/둘 다]
- tests/withrex_테스트_스크립트.md에서 "[기능명]" 관련 시나리오 기반

버그는 기록만 하고 수정하지 마.
```

---

## 커밋 안내

사용자에게 아래 두 시점에 커밋을 안내한다:

1. **Phase 2 완료 후** — 구현 커밋
   ```
   feat: [기능명] 구현
   ```

2. **Phase 3 완료 후** — 테스트 시나리오 커밋
   ```
   test: [기능명] 테스트 시나리오 추가
   ```
