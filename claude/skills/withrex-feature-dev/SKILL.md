# WITHREX Feature Development Skill

WITHREX 보안 스캐닝 제품의 신기능 개발 파이프라인.
**구현만 하고 끝내지 않는다.** 테스트 시나리오 생성과 QA 연동이 이 스킬의 핵심이다.

## 핵심 원칙

코드를 작성하기 전에 영향 범위를 파악하는 이유는, 나중에 발견하는 사이드 이펙트가 고치는 비용이 훨씬 크기 때문이다. 마찬가지로 테스트 시나리오를 구현과 함께 만드는 이유는, 기능을 막 완성한 시점에 엣지 케이스가 가장 잘 보이기 때문이다.

1. **영향 범위 먼저** — 코드 전에 어디에 영향을 주는지 파악한다
2. **구현과 테스트는 한 세트** — 기능을 만들면 시나리오도 같이 만든다
3. **기존 패턴 존중** — Phase 1에서 파악한 패턴을 이탈하지 않는다
4. **상태 추적** — 매 Phase 전환 시 `state_update.md`를 업데이트한다
5. **로그** — 새로운 동작이 추가된다면 모니터링 , 디버깅용의 로그를 `logs/` 폴더 아래 해당 서비스 명으로 로그를 남긴다
6. **동기화** — 새로고침 시 해당 기능의 상태가 복원돼야 한다

## 개발 파이프라인

### Phase 0: 요구사항 확인

시작 전에 아래 5가지를 명확히 한다:
1. 사용자가 원하는 기능이 정확히 무엇인가
2. 범위는 어디까지인가 (Backend만? Frontend도?)
3. 기존에 유사한 기능이 있는가
4. 어떤 모드에서 동작하는가 (스캔모드 / 취약점모드 / 둘 다)
5. 입력 유형에 따라 동작이 달라지는가

요구사항을 아래 형식으로 정리하고 사용자에게 확인받는다:

```markdown
## 요구사항 정리
- **기능명**: 
- **설명**: 
- **범위**: Backend / Frontend / 둘 다

### 모드 적용
- [ ] 스캔모드에서 동작하는가
- [ ] 취약점모드에서 동작하는가
- [ ] 모드별 동작 차이: 

### 입력 유형 영향
- [ ] 입력 유형과 무관한 기능인가
- [ ] 단일 IP / 다중 IP / 단일 대역 / 다중 대역에 따라 다른가 → 차이점:

### 사용자 시나리오
(사용자가 이 기능을 어떻게 사용하는지 한 문단으로)
```

### Phase 1: 영향 범위 분석

코드 작성 전에 반드시 수행한다. 이 단계를 건너뛰면 뒤에서 사이드 이펙트를 발견하고 되돌아오게 된다.

**탐색할 것:**
- 직접 수정이 필요한 파일 (새 파일 + 기존 수정 파일)
- 간접적으로 영향받는 파일 (같은 API, 같은 DB 테이블, 같은 WebSocket 이벤트)
- 기존 코드의 패턴 (API 라우트 방식, 에러 핸들링, 인증 체크, 응답 형식, 컴포넌트 구조)

결과를 `.claude/plan/plan_update.md`에 기록하고 사용자에게 검증받은 후 Phase 2로 넘어간다.
상세 문서 형식: `references/plan_template.md` 참조

### Phase 2: 구현

영향 범위 분석이 완료된 후에만 시작한다.

**순서:**
1. Backend (DB 스키마 → API 엔드포인트 → 비즈니스 로직 → 에러 핸들링)
2. Frontend (컴포넌트 → API 연동 → WebSocket 연동 → Arctic Recon 테마 스타일링)
3. 연동 확인 (curl로 API 응답 확인, 프론트에서 API 호출 확인)

파일 5개 변경할 때마다 `state_update.md`를 업데이트한다.
기존 기능에 영향을 줄 수 있는 변경을 발견하면 즉시 사용자에게 보고한다.

### Phase 3: 테스트 시나리오 생성

> 이 Phase가 이 스킬의 핵심이다. 구현 완료 후 반드시 수행한다.

구현한 코드를 분석해서 아래 관점으로 시나리오를 도출한다:
- 정상 동작 (Happy Path)
- 모드별 동작 (스캔모드 / 취약점모드가 다른 경우 각각)
- 입력 유형별 동작 (Phase 0에서 "무관"이면 대표 1개, 다르면 4가지 전부)
- 에러 케이스 (잘못된 입력, 권한 없음, 서버 에러, 타임아웃)
- 경계값 (빈 입력, 최대/최소값, IP 형식 경계)
- 기존 기능 영향 (Phase 1의 간접 영향 파일이 깨지지 않았는지)

시나리오는 `tests/withrex_테스트_스크립트.md`에 추가한다.
시나리오 형식 및 카테고리 코드: `references/test_scenario_guide.md` 참조

### Phase 4: 자체 검증

```
1. Backend — curl로 새 API 엔드포인트 응답 확인, 에러 케이스 확인, 인증 확인
2. Frontend — 페이지 로딩, API 호출, 기본 인터랙션 확인
3. 기존 기능 — Phase 1의 간접 영향 파일 기능이 살아있는지 빠르게 확인
```

발견한 버그는 즉시 수정하고 `.claude/bug/bug_update.md`에 기록한다.

### Phase 5: QA 연동

구현과 자체 검증이 완료되면 사용자에게 완료 보고서를 제출하고, QA 파이프라인 실행을 안내한다.

완료 보고서 형식 및 QA 파이프라인 호출 방법: `references/qa_handoff.md` 참조

## 상태 추적

`.claude/state/state_update.md`를 매 Phase 전환 시 업데이트한다.

```markdown
# Pipeline State

## 현재 상태
- **파이프라인**: Feature Dev
- **현재 Phase**: Phase N (단계명)
- **현재 작업**: 구체적인 작업 내용
- **마지막 업데이트**: YYYY-MM-DD HH:MM

## 진행률
- Phase 0 요구사항 확인: ✅ / 🔄 / ⬜
- Phase 1 영향 범위 분석: ✅ / 🔄 / ⬜
- Phase 2 구현: ✅ / 🔄 / ⬜
- Phase 3 테스트 시나리오 생성: ✅ / 🔄 / ⬜
- Phase 4 자체 검증: ✅ / 🔄 / ⬜
- Phase 5 QA 연동: ✅ / 🔄 / ⬜

## 컨텍스트 요약
- 기능: [기능명]
- 변경 파일: N개 / 새 파일: N개
- 영향받는 기존 기능: [목록]

## 다음 할 일
- [구체적인 다음 작업]
```

## 파일 구조

| 파일 | 경로 | 쓰기 방식 |
|------|------|-----------|
| 상태 추적 | `.claude/state/state_update.md` | 수시 업데이트 |
| 개발 계획 | `.claude/plan/plan_update.md` | 덮어쓰기 |
| 버그 기록 | `.claude/bug/bug_update.md` | 추가 기록 |
| 체크리스트 | `.claude/checklist/checklist_update.md` | 덮어쓰기 |
| 테스트 스크립트 | `tests/withrex_테스트_스크립트.md` | 기존 유지 + 추가 |

> `tests/withrex_테스트_스크립트.md`만 QA 파이프라인과 공유한다.
> 나머지는 `_update` 파일로 분리하여 QA 파이프라인의 기록과 충돌하지 않는다.
