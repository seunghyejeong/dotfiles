# TC Writer — 테스트 케이스 작성 스킬

## 목표

테스터가 **맥락 없이도** TC를 수행할 수 있는 수준으로 작성한다. 구체적으로:

- 진입 경로를 명시 (메뉴명 → 탭명 → 버튼명 순서)
- TC별 전제조건·사전 데이터 명시
- 기대결과를 검증 가능한 수준으로 명시 (에러 형태/위치/정확한 문구 포함)

---

## 입력 수용 방식

| 입력 유형 | 처리 방식 |
|-----------|-----------|
| 기능 명세서 / PRD | 명세에서 Happy Path + Edge Case를 도출해 TC 생성 |
| 화면 캡처 / 목업 이미지 | UI 요소(메뉴명, 버튼명, 필드명) 직접 추출 |
| 기존 TC 초안 (보완 요청) | 누락 항목(전제조건, 진입경로, 기대결과)만 보완 |
| 구두 설명만 | 추가 질문으로 UI 진입경로와 기대결과를 확인 후 작성 |

---

## TC 구조 (필수 필드)

각 TC는 반드시 아래 6개 필드를 포함한다.

```
TC ID     : [모듈번호]-[기능번호]-[순번]  예) 12-3-1
TC 명     : 기능을 한 문장으로 요약
전제조건  : TC 수행 전 반드시 준비되어야 하는 상태/데이터
입력 데이터 : 테스터가 직접 입력하거나 선택하는 값
상세 절차 : 진입경로 포함, 번호 매긴 단계별 액션
기대결과  : 에러 형태·위치·문구·화면 전환까지 명시
실제결과  : (테스터 기입란 — 빈칸)
Pass/Fail : (테스터 기입란 — 빈칸)
```

---

## 작성 규칙

### 1. 상세 절차 작성법

**절대 금지**: 진입경로 없는 최종 액션만 단독 기재
```
❌ 나쁜 예: "PDF 다운로드 클릭"
❌ 나쁜 예: "재시작"
```

**올바른 형식**: `[위치] > [위치] > [액션]` 흐름으로 단계별 기재
```
✅ 좋은 예:
1. 좌측 사이드바 > [보고서] 메뉴 클릭
2. 상단 탭 > [스캔 이력] 탭 선택
3. 목록에서 대상 스캔 행 클릭하여 상세 진입
4. 우상단 [PDF 다운로드] 버튼 클릭
```

### 2. 전제조건 작성법

사전 준비가 필요한 경우 반드시 명시. 유형별 예시:

| 유형 | 전제조건 예시 |
|------|--------------|
| 데이터 의존 | "스캔 이력 2건 이상 존재해야 함 (비교 기능 활성화 조건)" |
| 설정 의존 | "타임아웃 시간이 30초로 설정되어 있어야 함 (Settings > 스캔 설정)" |
| 권한 의존 | "Admin 계정으로 로그인된 상태" |
| 상태 의존 | "진행 중인 스캔이 없는 유휴 상태" |

전제조건이 없는 TC: `전제조건: 없음 (초기 상태에서 수행 가능)` 명시

### 3. 기대결과 작성법

에러/알림/화면 전환 모두 아래 요소를 포함:

| 요소 | 기재 내용 |
|------|-----------|
| **형태** | 인라인 에러 / 토스트 메시지 / 모달 팝업 / 화면 전환 |
| **위치** | 필드 하단 / 화면 우상단 / 화면 중앙 |
| **문구** | 정확한 텍스트 (따옴표로 표기) |
| **지속시간** | 자동 사라짐(3초) / 수동 닫기 필요 |

```
❌ 나쁜 예: "에러 표시됨"
❌ 나쁜 예: "로그인 실패 안내"

✅ 좋은 예: 
  - 비밀번호 입력 필드 하단에 인라인 에러 표시
  - 문구: "비밀번호가 올바르지 않습니다. 다시 확인해 주세요."
  - 필드 테두리 빨간색으로 변경, 수동으로 닫기 필요 없음
```

## 출력 형식

**기본: xlsx (Excel) 파일로 출력한다.** Markdown 요청 시에만 텍스트로 출력.

### xlsx 컬럼 구조 (순서 고정)

| 컬럼 | 설명 | 너비 기준 |
|------|------|-----------|
| TC-ID | `tc-001` 형식 또는 `[모듈]-[기능]-[순번]` | 좁음 |
| 제목 | 기능을 한 문장 요약 | 중간 |
| 우선순위 | P0 / P1 / P2 (P0 = 브랜드 컬러 강조) | 좁음 |
| 모드 | 공통 / 관리자 / 사용자 등 | 좁음 |
| 전제조건 | 사전 준비 상태/데이터 | 중간 |
| 테스트 데이터 | 테스터가 입력할 값 | 중간 |
| 수행 절차 | 진입경로 포함 번호 목록 | 넓음 |
| 기대 결과 | 형태·위치·문구 포함 | 넓음 |
| 실제 결과 | 테스터 기입란 (빈칸) | 중간 |
| Pass / Fail | 테스터 기입란 (빈칸) | 좁음 |

### 스타일 규칙 (openpyxl 사용)
- 헤더: 진한 네이비 배경 + 흰색 굵은 글씨
- 모듈 그룹 구분행: 중간 네이비 (병합 셀)
- 짝/홀 행 교번 배경으로 가독성 확보
- P0 우선순위: 브랜드 컬러(`#FE6F47`) 강조
- 헤더 틀 고정 + 자동 필터 적용
- 줄바꿈 허용, 행 높이 72pt 기본

## 작성 프로세스

### Step 1: 입력 분석 및 정보 수집

입력을 받으면:
1. TC 대상 기능/화면 파악
2. 부족한 정보 식별 후 한 번에 질문 (여러 번 끊지 않기)
3. 특히 아래 정보는 반드시 확인:
   - UI 진입경로 (어느 메뉴/탭에서 접근하는지)
   - 에러/알림 문구의 정확한 텍스트
   - 전제조건이 필요한 TC 여부

### Step 2: TC 범위 설계

기능 하나당 최소 3가지 케이스를 설계:
- **Happy Path**: 정상 입력 → 성공 케이스
- **Boundary Case**: 경계값, 최대/최소 입력
- **Error Case**: 잘못된 입력, 권한 없음, 데이터 없음

### Step 3: TC 작성

위 TC 구조 및 규칙을 적용하여 작성.  
기존 TC 보완 시: 누락 필드만 채우고 기존 내용 최대한 유지.

### Step 4: 품질 자가 검토

출력 전 아래를 확인:
- [ ] 모든 TC에 전제조건이 명시되어 있는가? (없으면 "없음"이라도 기재)
- [ ] 모든 상세 절차가 진입경로부터 시작하는가?
- [ ] 기대결과에 형태·위치·문구가 포함되어 있는가?
- [ ] TC ID가 일관된 형식인가?

## 자주 발생하는 실수 체크리스트

```
❌ 전제조건 없이 특정 데이터가 필요한 TC 작성
❌ "버튼 클릭" — 어느 화면의 버튼인지 미명시
❌ "에러 발생" — 형태/위치/문구 없음
❌ "성공" — 어떤 UI 변화가 성공 기준인지 미명시
❌ 동일 기능의 Happy Path만 작성하고 Edge Case 누락
```
