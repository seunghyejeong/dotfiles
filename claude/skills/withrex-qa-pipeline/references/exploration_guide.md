# 소스코드 탐색 가이드

Phase 1 계획 수립 시 아래 순서로 프로젝트를 탐색하여 시나리오를 도출한다.

## 1단계: 프로젝트 구조 파악

프로젝트 루트에서 디렉토리 구조를 확인한다.
- `backend/`, `server/`, `api/` 등 서버 측 코드 위치
- `frontend/`, `client/`, `src/` 등 클라이언트 측 코드 위치
- 설정 파일 (`requirements.txt`, `docker-compose.yml` 등)

## 2단계: Backend 탐색

| 탐색 대상 | 찾을 것 |
|----------|--------|
| API 엔드포인트 | 라우터/컨트롤러 파일의 모든 엔드포인트, HTTP 메서드, 파라미터, 응답 구조, 인증 필요 여부 |
| WebSocket 이벤트 | 핸들러의 모든 이벤트명과 메시지 구조, 서버↔클라이언트 양방향 구분 |
| DB 스키마 | 모델/스키마 정의, 테이블 관계(FK), 마이그레이션 파일 |
| 스캔/취약점 엔진 | 실행 로직(nmap 등 외부 도구 호출), 취약점 매칭 로직, 결과 파싱 및 저장 |

## 3단계: Frontend 탐색

| 탐색 대상 | 찾을 것 |
|----------|--------|
| 라우트/페이지 | 라우터 설정의 모든 페이지 경로, 주요 컴포넌트 구조 |
| 상태 관리 | 전역 상태 구조, 스캔 상태 머신 (idle → scanning → complete 등) |
| API 호출 | 프론트엔드에서 호출하는 모든 API, WebSocket 연결/구독 로직 |
| 사용자 인터랙션 | 폼 입력(IP, 모드), 버튼/액션(스캔 시작/중지), 실시간 업데이트 컴포넌트 |

## 4단계: 시나리오 도출

탐색 결과를 기반으로 아래 관점에서 시나리오를 도출한다:

1. 각 API 엔드포인트 → 정상/비정상 시나리오
2. 각 WebSocket 이벤트 → 수신/누락/재연결 시나리오
3. 각 페이지 → 로딩/표시/인터랙션 시나리오
4. 입력 유형 × 모드 조합 시나리오 (아래 매트릭스 참조)
5. 에러 케이스 → 잘못된 입력, 네트워크 끊김, 타임아웃
6. 엣지 케이스 → 빈 결과, 대량 결과, 동시 스캔

## 5단계: 입력 × 모드 매트릭스

|           | 스캔모드 | 취약점모드 |
|-----------|---------|-----------|
| 단일 IP   | ✓       | ✓         |
| 다중 IP   | ✓       | ✓         |
| 단일 대역  | ✓       | ✓         |
| 다중 대역  | ✓       | ✓         |

모든 조합이 필요하지는 않다. 입력 유형에 민감한 시나리오(파싱, 유효성 검증 등)만 전체 조합을 테스트하고, 나머지는 대표 입력 1~2개로 충분하다.
